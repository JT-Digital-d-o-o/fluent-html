# MVP Types

An MVP is not a cheaper product. It is the fastest way to get through the Build-Measure-Learn loop with the least effort. Pick the type that tests your riskiest assumption — not the one that feels most like "building."

---

## Smoke Test MVP

Test demand before building anything. A landing page, a "Buy Now" button, a fake door — if nobody clicks, you have your answer without writing a line of code.

### DO: Use smoke tests to validate the value hypothesis first

| Element | What to build | What to measure |
|---|---|---|
| Landing page | Problem headline + CTA | Signup/click-through rate |
| Fake door | Feature button that leads to waitlist | Click rate on the button |
| Pre-order page | Price + delivery promise | Conversion rate, willingness to pay |

Target: >5% conversion on a cold audience means real interest. <2% means the problem statement or positioning is wrong.

### DON'T: Treat low traffic as a failed test

A smoke test with 30 visitors proves nothing. Drive at least 200–300 targeted visitors before drawing conclusions. If you can't get traffic, your growth hypothesis has a problem too.

---

## Concierge MVP

Deliver the value manually to a handful of customers. No code, no automation — you are the product. This tests whether the solution actually solves the problem when executed perfectly.

### DO: Start with 3–5 customers and do everything by hand

1. Find 3–5 customers who match your target segment
2. Deliver the value manually (spreadsheets, emails, phone calls)
3. Measure: Do they come back? Do they refer others? Would they pay?

If manual delivery doesn't retain users, automating it won't help. If it does retain them, you now know exactly what to automate.

### DON'T: Skip concierge because it "doesn't scale"

The point is that it doesn't scale. You're buying learning, not building infrastructure. A concierge MVP that retains 5 customers tells you more than a coded product that acquires 500 and retains none.

---

## Wizard of Oz MVP

The customer sees a real-looking product, but behind the scenes a human does the work. Unlike concierge, the customer doesn't know it's manual. This tests the user experience and interface assumptions.

### DO: Use Wizard of Oz when the interface matters

Good fit when:
- The value depends on how the customer interacts with the product (recommendations, search, matching)
- You need to test whether users understand the flow before investing in automation
- The manual work per customer is under 15 minutes

### DON'T: Run Wizard of Oz longer than 2 weeks

This approach burns human hours fast. Set a hard deadline: 2 weeks or 20 customers, whichever comes first. By then you either have signal to build or signal to pivot.

---

## Single-Feature MVP

Build one thing that works end-to-end. Not a prototype, not a mockup — a real, usable product with exactly one feature that tests your core value hypothesis.

### DO: Ship the one feature that makes or breaks the value hypothesis

Strip everything away until one feature remains. That feature must be the one where, if users don't engage with it, the entire product fails. Ship it in production quality — half-broken UX tests your engineering, not your hypothesis.

### DON'T: Add "just one more feature" before launch

Every feature you add before testing the core hypothesis delays learning. The second feature can wait until the first one proves value. "Users would love it if we also..." is the sound of scope creep in MVP clothing.

---

## Choosing the Right MVP

### DO: Match MVP type to your riskiest assumption

| Riskiest assumption | MVP type | Time to signal |
|---|---|---|
| Nobody wants this | Smoke test | 1–3 days |
| The solution doesn't actually solve it | Concierge | 1–2 weeks |
| Users won't understand or use it | Wizard of Oz | 1–2 weeks |
| The core feature doesn't deliver value | Single-feature | 1 week (with AI-assisted dev) |

Start from the top. Don't build a single-feature MVP when a smoke test would answer the question faster.

### DON'T: Use MVP as an excuse for low quality

MVP means minimum viable — not minimum effort. A smoke test landing page should look professional. A single-feature MVP should work reliably. Users can't evaluate your value proposition through a broken experience.
