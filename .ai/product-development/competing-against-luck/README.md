# Competing Against Luck — Jobs to Be Done

Customers don't buy products. They hire them to make progress in a specific circumstance. Define the job first. Everything else — features, messaging, roadmap — follows from the job.

## Where This Fits

- **Before Lean Startup experiments**: use JTBD to write better hypotheses. "Users will complete onboarding faster" is weak. "Users hired this to feel confident before their first client call" is the hypothesis worth testing.
- **Inside Shape Up problem framing**: a properly articulated job IS a Shape Up problem statement. "Sharing multiple files takes too many steps" describes a circumstance + struggle — that's a job.
